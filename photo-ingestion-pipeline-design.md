# 写真取り込みパイプライン 設計書 v1.0
### お湯から、探す。 — フォトライブラリー画像の規約準拠インジェスト

最終更新: 2026-06-27 / 想定実装環境: Claude Code(ローカル or Cowork)

---

## 0. この設計の大原則(最重要)

**「収集(取得)」と「加工・登録」を分離する。自動化するのは後者だけ。**

調査の結果、自治体・観光連盟のフォトライブラリーは**いずれもスクレイピング(自動一括ダウンロード)を想定・許可していない**。多くが「観光振興目的に限る」「1目的1回」「申請制」といった条件を持つ。したがって:

- ❌ やらないこと: クローラーで画像を機械的に一括収集する
- ✅ やること: **規約に沿って人手で取得(DLボタン/申請)した画像**を、ローカルの `inbox/` に置き、そこから先(リサイズ・WebP変換・メタ付与・DB登録・クレジット紐付け)をスクリプトで全自動化する

この線引きにより、規約を一切破らずに「面倒な後処理」だけを自動化できる。自治体との関係を損なわないことは、本プロジェクトの資産そのものなので、ここは絶対に妥協しない。

### 判定ランクの意味
- **A(最優先・即実装)**: 申請不要・クレジット不要・継続利用可。立ち上げ時の主力。→ 旅東北・青森
- **B(条件付き可)**: 観光目的でWeb掲載可。クレジット要否を個別確認し、必要ならDBに保存して自動表示。
- **C(申請制)**: 人が申請→受領した画像を `inbox/` に投入。パイプラインは同じ。
- **D(除外)**: 「1DL1回」「再申請必須」「加工転用禁止」等、継続運用と両立しないものは**最初から使わない**。富山・猪苗代型は外す。

### 立ち上げ戦略への含意
**A+B だけで、東北全湯(14湯)+南九州(8湯)+大分(3湯)+長野(5湯)がカバーできる。** これは前回の調達台帳の「立ち上げは東北・南九州先行」と完全に一致。写真の取りやすさが、コンテンツの厚い地域と同じ方を指している。

## 2. データモデル(photoエンティティ)

取り込み時に**出典とクレジットと利用条件を必ず一緒に記録する**。これが後で「クレジット自動表示」「規約違反の検知」「ライセンス監査」を可能にする。

```
photo {
  id              string   // 例 "p_kusatsu_001"
  onsen_id        number   // 紐づく温泉ID (1-96)
  role            enum     // hero | card | scene | exterior | town
  source_type     enum     // library | facility | own | ugc
  source_name     string   // "旅東北" 等(判定表のライブラリー名)
  source_url      string   // 取得元ページURL
  terms_url       string   // 利用規約ページURL
  terms_rank      enum     // A | B | C  (Dは取り込まない)
  credit_required bool      // true なら表示必須
  credit_text     string   // "写真提供:旅東北" 等。不要なら空
  license_note    string   // "観光PR目的・加工不可" 等の自由メモ
  acquired_by     enum     // download | application
  acquired_at     date
  applied_no      string   // 申請制の受付番号など(C案件)
  orig_filename   string
  width           number
  height          number
  // 出力(生成物)
  webp_path       string   // 例 "/photos/kusatsu/p_kusatsu_001.webp"
  webp_w          number
  blurhash        string   // 低容量プレースホルダ(任意)
}
```

**運用ルール**: `credit_required=true` の写真は、UI側で必ず credit_text をオーバーレイ表示する(v9以降の `.credit` 要素がそのまま使える)。`terms_rank=C` は `applied_no` 必須。これを保存時にバリデーションする。

## 3. ディレクトリ構成(ローカル)

```
onsen-photos/
├── inbox/                 # ★人手で取得した原本をここに置く(規約準拠で取得)
│   ├── 旅東北/
│   │   ├── 銀山温泉_雪夜.jpg
│   │   └── ...
│   ├── 青森/
│   └── 霧島市/
├── manifest/
│   └── sources.yaml       # inboxの各画像→温泉ID・出典・規約の対応表(人が記入)
├── dist/                  # 出力(WebP・サムネ)。サイトにデプロイする成果物
│   └── kusatsu/
│       ├── p_kusatsu_001.webp
│       └── p_kusatsu_001.thumb.webp
├── db/
│   └── photos.json        # photoエンティティの集合(生成物)
└── scripts/
    ├── 01_validate.mjs    # manifest と inbox の整合チェック
    ├── 02_process.mjs     # リサイズ・WebP変換・blurhash
    ├── 03_register.mjs    # photos.json を生成/更新
    └── 04_report.mjs      # ライセンス監査レポート
```

`manifest/sources.yaml` は**人が書く唯一のファイル**。ここに「この画像はどの温泉の、どこから取った、どの規約の写真か」を記述する。例:

```yaml
- file: 旅東北/銀山温泉_雪夜.jpg
  onsen_id: 43
  role: hero
  source_name: 旅東北
  source_url: https://www.tohokukanko.jp/photos/...
  terms_url: https://www.tohokukanko.jp/photos/index.html
  terms_rank: A
  credit_required: false
  acquired_by: download
  acquired_at: 2026-07-01
```

## 4. パイプライン各ステップの仕様

### 01_validate.mjs — 取り込み前チェック
- `inbox/` の全ファイルが `sources.yaml` に記載されているか(漏れ検出)
- `sources.yaml` の各エントリの必須項目が埋まっているか
- `terms_rank: D` が混入していないか(混入したら**エラーで停止**)
- `terms_rank: C` なのに `applied_no` が無い場合は警告
- `onsen_id` が 1-96 の範囲か、実在するか
- 出力: 検証レポート(OK/NG一覧)。NGがあれば後続を実行しない

### 02_process.mjs — 画像処理(ここが自動化の主役)
- ライブラリ: `sharp`(Node)を使用
- 各画像について:
  - **リサイズ**: hero=幅1600px / card=幅800px / thumb=幅400px(アスペクト比維持)
  - **WebP変換**: quality 82、メタデータ(EXIF)は除去(プライバシー)
  - **トリミング**: 規約で「トリミングのみ可」の案件があるため、**自動クロップはしない**。サイズ変更(縮小)のみ。必要な構図調整は人が inbox 段階で行う
  - **加工制限の遵守**: フィルター・色調補正・合成は**一切しない**(猪苗代型規約に違反するため)。やるのは縮小とフォーマット変換だけ
  - blurhash 生成(任意・プレースホルダ用)
- 出力: `dist/<onsen>/` に WebP 各サイズ

### 03_register.mjs — DB登録
- `sources.yaml` + 処理結果(寸法等)から photoエンティティを生成
- `db/photos.json` にマージ(既存は id で上書き、新規は追加)
- credit_text を source_name から自動補完(例: 旅東北かつ credit_required=false → 空、焼津 → "写真提供:一般社団法人焼津市観光協会")
- 1温泉に複数写真がある場合の hero 優先順位を決定(role=hero を先頭に)

### 04_report.mjs — ライセンス監査
- 全 photo を terms_rank・source_name 別に集計
- credit_required=true の写真がUIでクレジット表示対象になっているか(設定の突き合わせ)
- 申請制(C)案件の applied_no 一覧(申請の記録として保管)
- カバレッジ: 96湯中、写真ありが何湯か。role=hero がある湯/ない湯のリスト
- 出力: `report.md`(Naomiさんが自治体とのやり取りや内部管理に使える)

## 5. サイトへの反映

- `db/photos.json` と `dist/` を、温泉サイト側のデータと結合
- 既存プロトタイプの `PHOTOS` 定数(現在はデモSVG)を、`photos.json` 由来の実画像参照に差し替え
- 写真が**ある湯はWebP表示、ない湯は現状の線画イラストのまま**(フォールバックは既に実装済み)。これにより、写真は集まった分だけ順次反映でき、一気に揃わなくてもサイトは常に成立する
- クレジット表示は photo.credit_required を見て自動。Aランク画像はクレジットなしで軽く、B/C案件は帯付き

## 6. 立ち上げ実行プラン(推奨順)

1. **Phase 1 — 東北・青森(Aランク)**: 旅東北 + Amazing AOMORI から、規約同意の上でDL。14湯+α。クレジット不要なので最も軽い。これだけでサイトの「写真がある率」が一気に上がる
2. **Phase 2 — 南九州・大分(Bランク)**: かごしまの旅・霧島市・おおいた風景写真集。クレジット要否を確認しDBに記録
3. **Phase 3 — 申請制(Cランク)**: 佐賀・福島など。前回提案した「利用相談の申請テンプレ」を使い、受領分を inbox へ
4. 並行して、施設提供(浴槽写真)の打診と、自社撮影(現地行脚)を進める

### 注意(Claude Code 実装時の禁止事項)
- フォトライブラリーへの自動アクセス・一括DLコードは**書かない**(規約違反)。スクリプトは必ず「ローカルの inbox を入力」とする
- 色調補正・フィルター・合成・自動クロップを実装しない(加工禁止規約への配慮)。許可されるのは縮小とWebP変換のみ
- EXIFの位置情報は出力時に除去する(撮影者プライバシー)
